## Теория

GraphQL – это язык запросов к API и одновременно среда выполнения этих запросов на стороне сервера. Он был предложен как альтернатива традиционным REST API в ситуациях, когда клиенту требуется гибко управлять составом получаемых данных.

В REST клиент обычно обращается к заранее определённым конечным точкам, например /tasks, /tasks/{id}, /users/{id}. Сервер возвращает фиксированный формат ответа, который определён логикой конкретного HTTP-обработчика. В GraphQL клиент отправляет один запрос на единый endpoint, а внутри запроса сам описывает, какие именно поля и в какой структуре он хочет получить.

Это позволяет решить две распространённые проблемы:
- over-fetching – сервер возвращает больше данных, чем реально нужно клиенту;
- under-fetching – для получения нужных данных приходится делать несколько отдельных запросов.

Таким образом, GraphQL удобен тогда, когда фронтенду или внешнему клиенту важно самостоятельно управлять выборкой полей и структурой ответа.

Основные элементы GraphQL

GraphQL API строится вокруг схемы. Схема определяет, какие типы данных существуют в системе, какие запросы может выполнять клиент и какие операции изменения данных допустимы.

Ключевые элементы GraphQL:

- type – описание объекта предметной области;
- input – входной тип для передачи данных в мутацию;
- Query – операции чтения данных;
- Mutation – операции изменения данных;
- resolver – функция на сервере, которая фактически выполняет запрос или мутацию.

### Содержание проекта

**Auth service** (порт 8081 +  gRPC порт 50051)
- Аутентификация пользователей
- Выдача токенов
- Проверка токенов через gRPC

**Tasks service** (порт 8082)
- CRUD для задач (TODO-список)
- Перед каждой операцией проверяет токен, отправляя gRPC-запрос в Auth service

**GraphQL service** (порт 8083)

- Альтернативный API поверх Tasks service
- Поддерживает запросы: получение списка задач, получение задачи по ID
- Поддерживает мутации: создание, обновление, удаление задач
- Использует общий сервисный слой с REST API
